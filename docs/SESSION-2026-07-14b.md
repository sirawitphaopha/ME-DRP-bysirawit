## 1. บั๊ก cross-browser — "เพื่อนกรอกผ่าน Safari ไม่เข้า เรากรอกผ่าน Chrome เข้า" (โจทย์หลัก)

**อาการ:** เพื่อนกรอกจาก Safari (มือถือ) → กดบันทึก รอนานแล้วขึ้น "บันทึกแล้ว" แต่ข้อมูลไม่ขึ้นคลังกลาง เครื่องอื่นไม่เห็น · Chrome ปกติ

**ไล่หาต้นเหตุ (ยืนยันจากของจริง):**
- Supabase ฝั่ง DB แข็งแรงดี — คอลัมน์ครบ, realtime เปิดถูก (publication + replica identity full), มีข้อมูลเข้าจริง, log เห็น POST 201 จากหลายเบราว์เซอร์
- log postgres เจอ `ERROR: invalid input syntax for type uuid` → คอลัมน์ `id` เป็นชนิด **uuid** รับเฉพาะรูปแบบ uuid
- ต้นตอในโค้ด (`components/MedDrpApp.tsx` `save`): `id: crypto.randomUUID ? crypto.randomUUID() : "r"+Date.now()` — เบราว์เซอร์ที่ **ไม่มี `crypto.randomUUID`** (Safari รุ่นเก่า / เปิดผ่านลิงก์ที่ไม่ใช่ https / เว็บวิวในแอป) ได้ `"r"+เลข` ที่ **ไม่ใช่ uuid** → Supabase ตีกลับ 400 → `save` ดัก `catch {}` เงียบ → ยังขึ้น "บันทึกแล้ว ✓" หลอก → ข้อมูลเข้าแค่ localStorage เครื่องนั้น

**แก้:**
- `lib/helpers.ts` → `uuid()` สร้าง UUID v4 เองได้ทุกเบราว์เซอร์ (ใช้ `crypto.randomUUID` ถ้ามี ไม่มีก็ `getRandomValues`/`Math.random` ประกอบเอง) + `isUuid()`
  - เทสแล้ว 3 เส้นทาง (Chrome / Safari เก่า / ไม่มี crypto) ได้ uuid v4 ถูกต้องหมด · Postgres cast `::uuid` ผ่าน
- `lib/data.ts` → `pushIncident(cfg, rec, timeoutMs=12000)` คืน boolean (true=สำเร็จหรือ 23505 มีอยู่แล้ว · timeout กันค้าง)
- `save`: ออก `id: uuid()` · เก็บ local ก่อน · ส่งแล้ว **ยืนยันผลจริง** → สำเร็จ = "บันทึกและส่งขึ้นระบบเรียบร้อย ✓" · ไม่สำเร็จ = เข้าคิว + toast เตือน (ไม่หลอกว่าสำเร็จอีก)

## 2. ระบบยืนยันการส่ง + คิวส่งซ้ำ + กู้รายงานเก่า

- คิว `PENDING_KEY = "meddrp_pending_v1"` · `enqueuePending` / `flushPending` (ออก uuid ใหม่ให้ id เก่าที่ผิด · 23505 = สำเร็จ) · state `pending`/`syncing`
- **แถบเตือนสีเหลืองบนสุด** เมื่อ `pending.length>0` — บอกจำนวนค้าง + ปุ่ม "ลองส่งอีกครั้ง" + แนะนำเปิดผ่าน Google Chrome
- flush อัตโนมัติตอน mount / `online` / `visibilitychange` (visible)
- `loadRecords`/`refreshRecords` เปลี่ยนจาก "ทับทิ้ง" → **"รวม"** (server + localOnly) · localOnly = `!serverIds.has(id) && (!isUuid(id) || pending)` — กู้รายงานเก่าที่ยังไม่ขึ้นระบบ + ไม่ดึงเคสในถังขยะกลับมา
- ⚠️ ข้อจำกัด: รายงานเก่าที่ **เคยรีเฟรชไปแล้ว** อาจถูกทับหายจาก cache ตั้งแต่ก่อนแก้ → กู้ไม่ได้ (ที่ยังค้างในเครื่องกู้ได้หมด)

## 3. ฟีเจอร์ลบรายงาน 2 ชั้น (ถังขยะ) — "ลบยาก ๆ"

พี่กัน approve mockup แล้วให้ทำทั้ง 2 แบบ (soft + hard)

- **DB:** `supabase/migrations/0007_soft_delete.sql` — คอลัมน์ `deleted_at timestamptz` (null=ใช้งาน / มีค่า=ถังขยะ) + index + RLS policy **delete** (เดิมมีแค่ insert/select/update) · **apply ขึ้น Supabase แล้ว**
- **`lib/data.ts`:** `fetchIncidents` กรอง `.is("deleted_at",null)` · เพิ่ม `fetchDeletedIncidents / softDeleteIncident / restoreIncident / hardDeleteIncident`
- **UI (`MedDrpApp.tsx`):**
  - ปุ่ม `🗑 ลบรายงาน` ในหน้ารายละเอียด (สีจาง แยกจากปุ่มแก้ไข กันเผลอกด)
  - ชั้น 1 ป๊อป `askDelete` "ย้ายไปถังขยะ" (amber · `doSoftDelete` ลบออกจาก records+cache แล้วโหลดถังขยะ)
  - หน้า "ตั้งค่า" (`renderManage`) เพิ่มส่วนถังขยะ (`state.trash` · `loadTrash` ตอน mount + เข้าหน้า) → ปุ่มกู้คืน (`doRestore`) / ลบถาวร
  - ชั้น 2 ป๊อป `hardTarget` (แดง) → **ต้องพิมพ์ HN ของเคสให้ตรง** ปุ่มถึงกดได้ (ไม่มี HN ให้พิมพ์ "ลบถาวร") · `doHardDelete`
  - รายงานในถังขยะไม่นับใน Dashboard (fetch กรองออก)
- **เทสจริงกับ DB:** insert → soft (active 0, trash 1) → restore (active 1) → hard (remaining 0) ✓

## 4. DRP เลือกระดับความรุนแรง A–I ได้ (บังคับกรอก)

**โจทย์:** "เคส DRP เลือก ME A–I ไม่ได้ ทั้งที่ควรเลือกได้" — เพราะ ME เป็นซับเซ็ตของ DRP (คุยกันเรื่อง ME ⊂ DRP) แต่ยังไม่ถึงขั้นรวม 2 หน้า แค่เพิ่มช่องความรุนแรงเข้า DRP

- แยกบล็อกความรุนแรงของ ME เป็นฟังก์ชันร่วม **`renderSeverityField()`** (เดิม inline) → เรียกทั้งฟอร์ม ME (ตำแหน่งเดิม) และ DRP (ถัดจาก "รายละเอียดเหตุการณ์/สาเหตุ")
- validation: ย้าย `if(!f.severity)` ขึ้นส่วนบังคับร่วม → **บังคับทั้ง ME + DRP**
- หน้ารายละเอียด + ประวัติ + โหมดแก้ไข ของ DRP: เพิ่มแถว/ช่องระดับความรุนแรง · หน้ารายงาน + ตาราง Dashboard เอา gate `type==="med"` ออก (โชว์ severity DRP)
- verify: build + ขับ Playwright จริง → ฟอร์ม DRP มีปุ่ม A–I ครบ 9 + เลือกได้

## 5. เลือกยา HAD → ติดธง High-alert อัตโนมัติ

**โจทย์:** คลังยามีธง HAD อยู่แล้ว เลือกยานั้นแล้วปุ่ม High-alert ควรติดเอง (พี่กันส่งรูป Warfarin (HAD) มา)

- `pickDrug()` เช็ค `d.had === true` → `setField("high_alert", true)` (ผู้ใช้ปลดเองได้ · ไม่ auto-off)
- verify: seed ยา Warfarin HAD → เลือกแล้วปุ่ม High-alert ติดอำพันเอง (Playwright)
